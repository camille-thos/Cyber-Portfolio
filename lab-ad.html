Phases

Mise en place d'une infrastructure Active Directory fonctionnelle dans un environnement réseau segmenté simulant un système d'information d'entreprise. Aucun mécanisme de détection ou SIEM n'est utilisé à ce stade.

Machines virtuelles

OPNsense — Firewall assurant la segmentation entre les réseaux. Interfaces : WAN (accès externe optionnel), LAN (réseau interne), OPT1 (réseau attaquant).

DC01 — Windows Server 2025 — Active Directory Domain Services, DNS. Domaine : lab.local

CLIENT01 — Windows 10 — Poste utilisateur joint au domaine, utilisé pour générer de l'activité légitime.

KALI01 — Machine utilisée pour simuler des attaques provenant d'un réseau distinct.

Organisation Active Directory

L'AD est structuré avec trois unités organisationnelles représentant les services de l'entreprise. Chaque OU contient un dossier Utilisateurs, Groupes et Ordinateurs. Cette organisation permet d'appliquer des GPO ciblées, de simuler des délégations d'administration et de reproduire des cas réalistes d'élévation de privilèges.

  • UO_Comptabilite
  • UO_IT
  • UO_RH

Validation

  • Le domaine Active Directory est opérationnel
  • Le client est joint au domaine
  • La communication réseau fonctionne via OPNsense
  • La segmentation réseau est en place

Mise en place de la journalisation de sécurité sur l'environnement Active Directory afin de rendre l'infrastructure observable avant toute simulation d'attaque. L'objectif est d'établir un comportement normal (baseline) du domaine pour permettre une analyse pertinente lors des phases suivantes.

Activation des stratégies d'audit

Les audits ont été configurés via une stratégie de groupe appliquée aux contrôleurs de domaine.

Default Domain Controllers Policy → Configuration ordinateur → Paramètres Windows → Paramètres de sécurité → Configuration avancée de la stratégie d'audit

Catégories activées

  • Ouverture de session : Succès / Échec
  • Gestion des comptes utilisateurs : Succès
  • Accès au service d'annuaire : Succès
  • Utilisation des privilèges sensibles : Succès / Échec

Établissement d'une baseline

Une activité légitime a été générée depuis CLIENT01 pour observer le comportement normal du domaine : ouverture de session, déconnexion/reconnexion, accès aux ressources, administration AD et trafic réseau standard. Cette étape permet de différencier ultérieurement une activité normale d'un comportement suspect.

Résultat

L'infrastructure produit désormais des journaux de sécurité exploitables. Le domaine est prêt pour la phase suivante : simulation d'attaques et analyse des traces générées.

Simulation d'une attaque par brute force sur le service RDP afin d'observer les traces générées dans les journaux Windows et valider la configuration d'audit mise en place lors de la phase précédente.

Contexte

RôleMachineIP
CibleDC01 — RDP TCP 3389192.168.2.2
AttaquantKALI01192.168.3.10

Étape 1 — Découverte du service

Vérification de l'exposition du service RDP depuis le réseau attaquant. Résultat : port 3389 ouvert, service ms-wbt-server détecté.

Étape 2 — Simulation du brute force

Multiples tentatives d'authentification générées depuis Kali. Résultats : multiples tentatives échouées, puis découverte d'un mot de passe valide.

Étape 3 — Analyse des journaux Windows

Analyse via Observateur d'événements → Journaux Windows → Sécurité. Comportement identifié : succession rapide d'échecs d'authentification depuis la même IP, suivie d'une connexion réussie.

Event IDDescription
4625Échec de connexion
4624Connexion réussie
4776Validation des identifiants

Indices détectables

  • Grand nombre d'Event ID 4625 en peu de temps
  • Même compte ciblé, même source réseau
  • Réussite finale après plusieurs échecs
  • Corrélation temporelle permettant d'identifier une attaque automatisée

Conclusion

L'environnement permet d'identifier une tentative de brute force, d'analyser les journaux Windows et de préparer la mise en place d'une réponse automatisée — objet de la Phase 04.

Déploiement d'un SIEM/XDR open source sur l'infrastructure existante afin d'automatiser la détection des comportements suspects et de déclencher des réponses sans intervention manuelle.

Règles de détection

  • Brute force RDP — plus de 5 Event ID 4625 en moins de 60 secondes depuis la même IP → alerte niveau 10
  • Authentification réussie après échecs — Event ID 4624 immédiatement après une série de 4625 → alerte niveau 12
  • Création de compte hors baseline — Event ID 4720 depuis un compte non attendu → alerte niveau 8

Réponse automatisée

  • Blocage IP — déclencheur niveau ≥ 10 : l'IP source est bloquée automatiquement via firewall-drop
  • Désactivation de compte — déclencheur niveau ≥ 12 : script PowerShell déclenché sur DC01 via l'agent, le compte compromis est suspendu
Capture — IP 192.168.3.10 bloquée automatiquement
Capture — Compte désactivé dans AD après déclenchement automatique

Validation

Le scénario de la Phase 03 est rejoué pour vérifier la chaîne complète : KALI01 lance le brute force → Wazuh détecte → active response déclenche → IP bloquée → alerte visible dans le dashboard.

Capture — Timeline des alertes Wazuh lors du rejeu du scénario

Limites et pistes d'amélioration

  • Seuils calibrés pour le lab — à ajuster en production selon la baseline réelle
  • La désactivation de compte peut générer des faux positifs — une couche SOAR serait nécessaire en production
  • Piste suivante : intégrer Sysmon pour détecter des techniques plus avancées (Pass-the-Hash, Kerberoasting)
Objectifs atteints
  • ✓ Comprendre les techniques d'attaque courantes en environnement Active Directory
  • ✓ Analyser les journaux Windows et identifier des indicateurs d'activité suspecte
  • ✓ Mettre en place des mécanismes de détection adaptés
  • ✓ Appliquer des mesures de durcissement cohérentes
  • ✓ Automatiser la réponse aux incidents via Wazuh Active Response
  • ✓ Documenter une démarche d'amélioration continue
